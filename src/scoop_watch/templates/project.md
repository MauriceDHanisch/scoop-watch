# Project description

Replace this file with a precise description of your research project. The
synthesis agent reads it to decide whether each arXiv paper overlaps with your
work, so be specific and concrete — vague descriptions produce vague briefings.

The fastest way to write a good version of this file is:

    scoop-watch author <project>

which launches an agent that interviews you and writes it for you.

## What this file should contain

- **The problem.** What question does the project answer? What is the standard
  approach, and why is it insufficient?
- **The method.** What does the project actually do — models, data, training
  setup, key equations. Enough detail that the agent can recognize a competing
  approach from an abstract alone.
- **The novel contribution.** What is genuinely new? This is what a "scoop"
  would overlap with. State it explicitly.

## Sub-themes

If the project has distinct parts, give each one a `## Theme:` heading. The
briefing layout groups scoop candidates under these themes, so name them
clearly. For example:

## Theme: <short name of the application or problem>

<one or two paragraphs>

## Theme: <short name of the method or architecture>

<one or two paragraphs>

Keep the whole description focused — a few well-written paragraphs per theme
beat a long unstructured dump.
